# Multi-Participant Video Consultation

**Date:** 2026-04-20
**Status:** Approved
**Scope:** Couple + family therapy (2-5 patients + 1 psy)

---

## Context

The current video consultation system supports only 1:1 sessions (1 psychologist + 1 patient). Psychologists need to conduct couple therapy, family therapy, and small group sessions remotely. LiveKit (self-hosted on OVH VPS) natively supports multi-participant rooms at no additional cost.

## Requirements

- Support 2-5 patients in a single video session (+ 1 psy = 3-6 total)
- All participants must be existing patients of the psychologist
- One patient is designated as "primary" (for billing/invoice purposes)
- Each participant receives their own video join token/link
- Existing 1:1 flow remains unchanged (backward compatible)
- Plan gating: Pro/Clinic only (same as current video)
- GDPR consent required for each individual participant before joining

## Decisions

| Decision | Choice | Rationale |
|---|---|---|
| Architecture | Table pivot `appointment_participants` | Backward compatible, minimal impact on existing queries |
| Patient principal | `appointment.patientId` | Keeps billing logic intact |
| Participant limit | Max 5 patients (validated server-side) | VPS resource constraint + UX quality |
| Billing | Single invoice to primary patient, total session fee | Simplest; matches real-world couple/family billing |
| Session creation on end | 1 session linked to primary patient | Other participants stored in `participantIds` column |
| Session type | `group` (online implied by `isOnline` + VideoRoom presence) | Reuses existing enum, no new value needed |
| LiveKit maxParticipants | Set to 6 for group rooms | Safety limit on self-hosted VPS |

## Database Schema

### New table: `appointment_participants`

```prisma
model AppointmentParticipant {
  id             String    @id @default(uuid())
  appointmentId  String    @map("appointment_id")
  patientId      String    @map("patient_id")
  videoJoinToken String?   @unique @map("video_join_token")
  videoLinkSentAt DateTime? @map("video_link_sent_at")
  joinedAt       DateTime? @map("joined_at")
  createdAt      DateTime  @default(now()) @map("created_at")

  appointment Appointment @relation(fields: [appointmentId], references: [id], onDelete: Cascade)
  patient     Patient     @relation(fields: [patientId], references: [id], onDelete: Cascade)

  @@unique([appointmentId, patientId])
  @@index([patientId])
  @@map("appointment_participants")
}
```

### New column on `Session`: `participantIds`

```prisma
model Session {
  // ... existing fields ...
  participantIds String[] @default([]) @map("participant_ids")
}
```

Stores the UUIDs of all additional participants (beyond the primary `patientId`). Allows querying "all sessions where patient X participated" via `@> ARRAY[patientId]`.

### SessionType enum — No change needed

`group` already exists in the enum (`individual | group | online`). No migration required.

### Relations

- `Appointment` gains: `participants AppointmentParticipant[]`
- `Patient` gains: `appointmentParticipations AppointmentParticipant[]`

### VideoRoom — No schema change

The `patientJoinedAt` field is updated on the first participant join (primary OR secondary). This maintains backward compatibility with existing status logic in `getTodayRooms()`.

## Backend Changes

### 1. New endpoint: Create group appointment

```
POST /appointments/group
```

**Guards:** `@UseGuards(KeycloakGuard, RolesGuard, SubscriptionGuard)` + `@RequirePlan(PRO, CLINIC)` + `@Roles('psychologist', 'admin')`

> This endpoint lives in the appointments controller. `BillingModule` must be imported in `AppointmentsModule` for `SubscriptionGuard` DI.

**DTO:**
```typescript
class CreateGroupAppointmentDto {
  @IsUUID() patientId: string;          // Primary patient (billed)
  @IsArray() @IsUUID(undefined, { each: true })
  @ArrayMinSize(1) @ArrayMaxSize(4)
  participantIds: string[];              // Additional participants (1-4)
  @IsDateString() scheduledAt: string;
  @IsInt() @Min(15) @Max(180) duration: number;
  @IsOptional() @IsUUID() consultationTypeId?: string;
}
```

**Validation:**
- All patients (primary + participants) must belong to `psychologistId`
- No duplicate patientIds
- `participantIds` does not contain `patientId`
- Total participants: 2-5 (primary + 1-4 additionals)

**Behavior:**
- Creates appointment with `isOnline: true`
- Primary patient's `videoJoinToken` is generated by existing appointment creation logic
- Creates `AppointmentParticipant` rows with individual `videoJoinToken` per additional participant
- Sends video link email to all participants (primary + additionals)
- Participants without email: no email sent, API response includes `participantsWithoutEmail: string[]` warning
- Audit log: `action: 'CREATE'`, `entityType: 'appointment'`, `metadata: { participantIds, primaryPatientId, isGroup: true }`

### 2. Modified: `generatePatientToken(joinToken)`

**New logic:**
1. First check `appointment.videoJoinToken` — if match, generate token for primary patient (existing behavior)
2. If no match, check `appointment_participants.video_join_token` — if match, generate token for that participant
3. **GDPR consent check** applies identically to all participants: lookup `gdprConsent` by the joining participant's `patientId` (not `appointment.patientId`)
4. LiveKit token: `identity: 'patient-{patientId}'`, `name: patient.name` (display name from DB)
5. On first join: update `appointment_participants.joined_at`
6. On first ANY participant join: update `VideoRoom.patientJoinedAt` (if not already set)

### 3. Modified: `recordConsent(joinToken)`

**New logic:**
1. Check `appointment.videoJoinToken` first — if match, consent for primary patient (existing)
2. If no match, check `appointment_participants.video_join_token` — if match, consent for that participant's `patientId`

### 4. Modified: `createRoom()`

For group appointments (has participants):
```typescript
await this.roomService.createRoom({
  name: roomName,
  emptyTimeout: 600,       // 10min (vs 5min for 1:1) — allows async family joining
  maxParticipants: 6,      // 5 patients + 1 psy safety limit
});
```

For standard 1:1 appointments (no change):
```typescript
await this.roomService.createRoom({ name: roomName, emptyTimeout: 300 });
```

### 5. Modified: `endRoom()`

- Creates 1 session with `type: 'group'` (if appointment has participants)
- Session linked to primary patient (`appointment.patientId`)
- `participantIds` column stores all additional participant UUIDs
- Duration calculated from psy join time (unchanged)
- `paymentAmount` on the appointment = total session fee (billed to primary patient)

### 6. Modified: `getTodayRooms()`

- Include participant count in response
- New field: `participantCount: number` (1 for regular, 2-5 for group)
- New field: `participantsJoined: number` (how many have joined)
- Status logic: if `VideoRoom.patientJoinedAt` is set (by ANY participant), status = `patient_waiting`

### 7. Email notifications

- When group appointment is created, send video link to each participant who has an email
- Each email contains the participant's unique join token URL
- Reuse existing video invitation email template with adjustment: "Consultation de groupe" wording
- Participants without email are skipped (psy shares link manually)

## Frontend Changes

### 1. Create Appointment Dialog

When creating a new appointment with `isOnline: true`:
- Show "Ajouter des participants" button below patient selector
- Opens a multi-select dropdown of the psy's patients (excluding already-selected primary)
- Shows selected participants as chips/badges (max 4)
- Badge on the dialog showing total participant count
- Calls `POST /appointments/group` instead of regular `POST /appointments` when participants > 0

### 2. VideoLayout (video-room.tsx)

**Current:** 1 remote track full-screen + local PiP.

**New adaptive grid:**
- 1 remote: full-screen (unchanged, backward compatible)
- 2 remotes: side-by-side (50%/50%)
- 3 remotes: 2 top + 1 bottom centered
- 4 remotes: 2x2 grid
- 5 remotes: 3 top + 2 bottom

**Implementation:**
- Use `useTracks` to get all remote camera tracks
- Dynamic grid CSS with `grid-template-columns` / `grid-template-rows`
- Participant name overlay on each tile (from LiveKit token `name` property)
- Local (psy) remains as PiP overlay bottom-right
- Notes panel on the right (unchanged)

### 3. Today's rooms list

- Show participant count badge: "2 patients", "Famille (4)"
- Show how many have joined: "2/4 connectes"

### 4. Patient join page

- No change needed — each patient joins via their unique token
- They see all other participants in the grid automatically (LiveKit handles this)
- GDPR consent screen applies individually per participant (existing flow)

## Edge Cases

| Scenario | Handling |
|---|---|
| Participant token used after room ended | 400: "Cette consultation est terminee" (existing) |
| Primary patient doesn't join but others do | Room functions normally; `patientJoinedAt` set by first joiner; session still linked to primary |
| Psy ends call while participants still connected | LiveKit room deleted, all participants disconnected |
| One participant leaves mid-session | Others continue; no special handling needed |
| Duplicate patient in participantIds | 400 validation error |
| Patient not owned by this psy | 403 forbidden |
| Patient deleted (GDPR purge) after group appointment created | `onDelete: Cascade` removes participant row; their token becomes invalid; room functions for others |
| Participant without email address | No email sent; included in `participantsWithoutEmail` response; psy shares link manually |
| Modification of participant list after creation | Out of scope — psy must cancel and recreate the appointment |

## Security & Compliance

- **Multi-tenant isolation:** All patients validated against `psychologistId` at creation time
- **GDPR consent:** Each participant must independently consent to video consultation before receiving a token
- **Audit logging:** Group appointment creation logged with all participant IDs
- **Token isolation:** Each participant has their own unique, unguessable token (UUID v4)
- **Plan gating:** `POST /appointments/group` requires Pro/Clinic subscription (same as video room creation)

## Migration Plan

1. Add `appointment_participants` table (non-breaking, new table)
2. Add `participantIds String[]` column to `sessions` table (non-breaking, default `[]`)
3. Add relations to Prisma schema (`Appointment.participants`, `Patient.appointmentParticipations`)
4. Import `BillingModule` in `AppointmentsModule` for `SubscriptionGuard` DI
5. Deploy backend with new endpoint + modified token logic
6. Deploy frontend with multi-select and grid layout
7. Existing 1:1 appointments continue working without any change

---

## Out of Scope

- Group therapy with 10+ participants
- External invitees (non-patients)
- Per-participant billing
- Modification of participant list after creation (cancel + recreate)
- Screen sharing (future enhancement)
- Recording (future enhancement, GDPR implications)
- In-person group sessions (use same `type: 'group'`, no VideoRoom)
